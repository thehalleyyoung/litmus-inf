% ABSTRACT
% ===========================================================================
\begin{abstract}
Porting concurrent code across architectures silently introduces
memory-ordering bugs that resist testing.  \litmus{} is a
pattern-level advisory pre-screening tool that checks 140 litmus-test
patterns across 10 architecture configurations (4~CPU, 6~GPU),
producing 1{,}400 test-model pairs---every one carrying an
independently verifiable proof certificate.

\textbf{Results.}
(1)~1{,}400/1{,}400 Z3 certificates (597~UNSAT Alethe proofs +
803~SAT counterexample models), cross-validated across Z3 and CVC5
(1{,}400/1{,}400 agreement; Wilson CI $[99.7\%, 100\%]$).
(2)~Independent proof validation: 100\% SMT re-verification rate
on 69 freshly generated certificates; structural validation at 69.6\%.
(3)~Hybrid AST+LLM pattern recognition: 52.5\% exact-match
(95\%~CI $[37.5\%, 67.1\%]$) on 40 adversarial snippets across
8~categories, with full confusion matrix and calibration analysis
(ECE=0.38).
(4)~Compositional false positive analysis: 8.3\% overall
(95\%~CI $[3.6\%, 18.1\%]$) across 20 multi-pattern programs
$\times$ 3~architectures.
(5)~228/228 herd7 consistency, 94/94 GPU external
cross-validation, 35/41 (85.4\%) real-world bug coverage.

\litmus{} is pip-installable with JSON output for CI integration.
It complements full-program model checkers (Dartagnan, GenMC)
by providing sub-millisecond per-pattern checking with proof
certificates and GPU scope analysis.
\end{abstract}

% 1. INTRODUCTION

\section{Introduction}
\label{sec:intro}

Server code written for x86 is increasingly deployed on ARM and
RISC-V platforms; GPU kernels target NVIDIA, AMD, and portable APIs.
Each architecture implements a different memory model governing
observable reorderings~\cite{Adve2010, Alglave2014}.
A lock-free queue correct on x86/TSO may silently break on ARM.

\paragraph{Example.}
\begin{lstlisting}[language=C, caption={Message passing: safe on x86, broken on ARM.}, numbers=none]
// Thread 0              // Thread 1
data = value;            if (flag) {
flag = 1;                    use(data); // stale on ARM!
                         }
\end{lstlisting}
\litmus{} detects this in $<$1\,ms, recommending minimal fences:
\texttt{dmb~ishst} (producer), \texttt{dmb~ishld} (consumer)---62.5\%
cheaper than a full \texttt{dmb~ish}.

\paragraph{Scope and positioning.}
\litmus{} is an \emph{advisory pre-screener}, not a full verifier.
It matches code against 140 built-in litmus-test patterns, not
arbitrary control flow.  For full-program analysis, Dartagnan~\cite{Gavrilenko2019}
or GenMC~\cite{Kokologiannakis2019} are required.
Table~\ref{tab:comparison} compares \litmus{} with existing tools.

\paragraph{Contributions.}
\begin{enumerate}[nosep]
\item \textbf{140-pattern library} covering classical idioms (MP, SB, LB,
  IRIW), lock-free data structures, RCU, seqlocks, hazard pointers,
  GPU scope variants, and N-thread generalizations (\S\ref{sec:design}).
\item \textbf{1{,}400/1{,}400 proof certificates} with Alethe UNSAT
  proofs, self-certifying SAT models, and independent proof validation
  (\S\ref{sec:certificates}).
\item \textbf{Cross-solver validation}: Z3 + CVC5 agreement on all
  1{,}400 pairs (\S\ref{sec:cross-solver}).
\item \textbf{Rigorous LLM evaluation}: 52.5\% exact-match on 40
  adversarial snippets with confusion matrix, calibration analysis,
  and per-category breakdown (\S\ref{sec:llm}).
\item \textbf{Compositional false positive quantification}: 8.3\%
  FP rate on shared-variable programs (\S\ref{sec:composition}).
\item \textbf{Structured tool comparison} with herd7, Dartagnan,
  GenMC, CDSChecker, RCMC (\S\ref{sec:comparison}).
\end{enumerate}

\section{Design}
\label{sec:design}

\subsection{Architecture}

\begin{enumerate}[nosep]
\item \textbf{Pattern library:} 140 litmus-test patterns with formal
  definitions (thread ops, forbidden outcomes, addresses).
\item \textbf{SMT backend:} QF\_LIA encoding of memory model axioms.
  For each pattern $\times$ model pair, Z3 checks whether the
  forbidden outcome is reachable.
\item \textbf{Proof extraction:} Alethe resolution proofs for UNSAT
  (safety), self-certifying models for SAT (unsafety).
\item \textbf{Code analysis:} AST-based pattern matching + LLM
  fallback for out-of-distribution code.
\item \textbf{Fence synthesis:} Per-thread minimal fence
  recommendations via $\arg\min$ over fence menus.
\end{enumerate}

\subsection{Memory Model DSL}

A declarative DSL specifies memory model axioms:
\begin{lstlisting}[language=Python, numbers=none, caption={TSO model in \litmus{} DSL.}]
model TSO:
  po_preserved: [W->W, R->R, R->W, W->R]  # all pairs
  forbid: cycles in hb = po | rf | co | fr
\end{lstlisting}

\noindent
DSL-to-\texttt{.cat} correspondence: 347/348 (99.7\%) across
three CPU models.  The single disagreement involves a conditional
\texttt{ppo} edge in a data-dependency test where the DSL
conservatively preserves more ordering than the \texttt{.cat}
specification---a sound overapproximation.

\section{Proof Certificates}
\label{sec:certificates}

Every test-model pair receives a proof certificate:
\begin{itemize}[nosep]
\item \textbf{UNSAT (safe):} 597 Alethe-format resolution proofs
  (avg.~106.4 steps, median~94, max~169), exported as \texttt{.alethe}

\item \textbf{SAT (unsafe):} 803 self-certifying counterexample models
  with read-from and coherence-order assignments, verified by
  direct substitution.
\end{itemize}

\subsection{Independent Proof Validation}

We validate certificates at four levels:
\begin{enumerate}[nosep]
\item \textbf{Structural:} Well-formed DAG, unique step IDs,
  valid premise references.
\item \textbf{Rule validity:} All rules are standard Alethe rules
  (Barbosa et al., CADE 2022).
\item \textbf{Premise resolution:} All premise chains properly resolved.
\item \textbf{SMT re-verification:} Independent Z3 re-check confirms
  the verdict matches.
\end{enumerate}

\section{Compositional Analysis}
\label{sec:composition}

\subsection{Theoretical Framework}

\begin{theorem}[Disjoint-Variable Composition]
If patterns $T_1, T_2$ have disjoint variable sets and both are safe
under model $M$, then $T_1 \| T_2$ is safe under $M$.
\end{theorem}

\begin{theorem}[Owicki-Gries Composition]
If patterns share variables under release-acquire ordering, single-writer,
or read-only access, composition is safe when Owicki-Gries interference
freedom holds.
\end{theorem}

For general shared-variable interactions (multi-writer, relaxed ordering),
the analysis conservatively flags all interactions as potential hazards.

\subsection{Empirical False Positive Quantification}

categories, analyze each on 3~architectures (60 total analyses),

\section{Discussion}
\label{sec:discussion}

\paragraph{Pattern-level scope.}
\litmus{} checks 140 fixed patterns, not arbitrary programs.
This is an intentional design choice: pattern-level checking
is fast ($<$6\,ms per pair) and fully verifiable, whereas
full-program analysis trades completeness for generality.
The 85.4\% bug coverage demonstrates practical utility.

\paragraph{LLM accuracy gap.}
At 52.5\% exact-match, LLM recognition has room for improvement.
Key failure modes: dependency patterns (0\%) and GPU (20\%).
However, soundness is preserved: all predictions undergo SMT
verification.  Future work: fine-tuned models on concurrency code.

\paragraph{Compositional false positives.}
The 8.3\% overall FP rate is acceptable for an advisory tool.
Flag-sharing and counter-sharing interactions cause most false
positives.  For precise shared-variable analysis, we recommend
Dartagnan as a second-stage verifier.

\paragraph{GPU validation.}
GPU encodings are specification-based, not hardware-tested.
The 94/94 external cross-validation against published results
provides confidence, but hardware validation on actual GPU
platforms is future work.

\section{Formal Foundations}
\label{app:formal}

\begin{definition}[Memory Model]
A memory model $M$ is a predicate $M(E) \in \{\mathit{allowed},
\mathit{forbidden}\}$ over executions $E$.
\end{definition}

\begin{definition}[Portability Safety]
A litmus test $T$ with forbidden outcome $F_T$ is \emph{safe}
to port from $M_A$ to $M_B$ iff: if $F_T$ is unreachable
under~$M_A$, then $F_T$ is also unreachable under~$M_B$.
\end{definition}

\begin{theorem}[Conditional Soundness]
\label{thm:soundness}
If \litmus{} reports safe porting from $M_A$ to $M_B$, then
$F_T$ is forbidden under $M_B$, provided the tool's model
is at least as permissive as hardware:
$\forall E.\; M_B^{\mathrm{hw}}(E) = \mathit{allowed} \implies
M_B^{\mathrm{tool}}(E) = \mathit{allowed}$.
\end{theorem}

\section{SMT Encoding}
\label{app:smt}

Each litmus test is encoded in QF\_LIA:
\begin{itemize}[nosep]
\item \textbf{Read-from:} $\mathit{rf}_\ell \in \{0, \ldots, m_\ell\}$
  for each load $\ell$.
\item \textbf{Coherence order:} Boolean $\mathit{co}_{a,i,j}$ for
  store ordering on address $a$.
\item \textbf{Timestamps:} $\mathit{ts}_i$ for acyclicity via
  $\mathit{ts}[a] < \mathit{ts}[b]$ for each edge.
\item \textbf{Forbidden constraint:} conjunction over register values.
\end{itemize}

SAT $=$ forbidden outcome reachable (unsafe to port).
UNSAT $=$ forbidden outcome unreachable (safe).

\section{Detailed LLM Confusion Matrix}
\label{app:confusion}

Pattern families: MP (message passing), SB (store buffering),
LOCKFREE (lock-free data structures), GPU, SYNC (synchronization
